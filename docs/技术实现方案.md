## 4. action.yml 定义

```yaml
name: 'Blog Global Translator'
description: 'Auto-translate GitHub Pages blog posts via AI'
inputs:
  ai_provider:
    description: 'openrouter | siliconflow | custom'
    required: true
  ai_api_key:
    description: 'AI provider API key'
    required: true
  ai_model:
    description: 'Model name (e.g. google/gemini-2.5-flash)'
    required: true
  ai_base_url:
    description: 'Custom AI base URL (only when ai_provider=custom)'
    required: false
  base_lang:
    description: 'Source language code'
    required: true
  target_langs:
    description: 'Comma-separated target language codes'
    required: true
  github_token:
    description: 'GitHub Token for PR creation (GITHUB_TOKEN or GitHub App token)'
    required: false
    default: ${{ github.token }}
runs:
  using: 'node20'
  main: 'dist/index.js'
```

**AI Provider 预设 Base URL**：


| provider    | base_url                        |
| ------------- | --------------------------------- |
| openrouter  | `https://openrouter.ai/api/v1`  |
| siliconflow | `https://api.siliconflow.cn/v1` |
| custom      | 用户通过`ai_base_url` 指定      |

## 5. GitHub 认证方案

支持两种认证方式，`github_token` 输入接受任意一种 token：

### 方案 A：GITHUB_TOKEN（默认）

使用 workflow 自动提供的 `GITHUB_TOKEN`，零配置。

- 优点：无需额外配置
- 限制：token 权限绑定当前仓库，PR 提交者显示为 `github-actions[bot]`，无法触发其他 workflow

### 方案 B：GitHub App（推荐）

创建一个 GitHub App，用 `actions/create-github-app-token@v3` 生成 installation token，独立于用户身份运行。

**优势**：

- 独立于用户登录，bot 身份运行
- 精细权限控制（只授予 contents:write + pull-requests:write）
- PR 提交者显示为 App 的 bot 账号（如 `blog-global-bot[bot]`）
- App 创建的 PR 可以触发其他 workflow（GITHUB_TOKEN 创建的不行）
- 可跨仓库使用（同一个 App 安装到多个仓库）

**用户配置步骤**：

1. 在 GitHub Settings → Developer settings → GitHub Apps 创建 App
2. 权限设置：`Contents: Read & Write`、`Pull requests: Read & Write`
3. 安装到目标仓库
4. 将 App Client ID 存为仓库 Variable（`APP_CLIENT_ID`）
5. 将 App Private Key 存为仓库 Secret（`APP_PRIVATE_KEY`）

## 6. 用户 Workflow 示例

### 使用 GITHUB_TOKEN（简单）

```yaml
name: Blog Global Translate
on:
  push:
    branches: [main]
    paths: ['_posts/**']

permissions:
  contents: write
  pull-requests: write

jobs:
  translate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0
      - uses: your-name/blog-global@v1
        with:
          ai_provider: openrouter
          ai_api_key: ${{ secrets.AI_API_KEY }}
          ai_model: google/gemini-2.5-flash
          base_lang: zh-CN
          target_langs: en,ja,ko
```

### 使用 GitHub App（推荐）

```yaml
name: Blog Global Translate
on:
  push:
    branches: [main]
    paths: ['_posts/**']

jobs:
  translate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/create-github-app-token@v3
        id: app-token
        with:
          client-id: ${{ vars.APP_CLIENT_ID }}
          private-key: ${{ secrets.APP_PRIVATE_KEY }}
          permission-contents: write
          permission-pull-requests: write
      - uses: actions/checkout@v6
        with:
          token: ${{ steps.app-token.outputs.token }}
          fetch-depth: 0
      - uses: your-name/blog-global@v1
        with:
          ai_provider: openrouter
          ai_api_key: ${{ secrets.AI_API_KEY }}
          ai_model: google/gemini-2.5-flash
          base_lang: zh-CN
          target_langs: en,ja,ko
          github_token: ${{ steps.app-token.outputs.token }}
```

## 7. 核心模块详细设计

### 7.1 框架检测器 (detector.ts)

MVP 仅支持 Jekyll，但检测器设计为可扩展。

**Jekyll 检测逻辑**：

1. 存在 `_config.yml`
2. 存在 `_posts/` 目录
3. 排除 Hexo（Hexo 也有 `_config.yml`，但同时有 `source/_posts/` 和 `package.json` 含 hexo 依赖）

**检测结果接口**：

```typescript
interface DetectResult {
  framework: 'jekyll' | 'hugo' | 'hexo' | 'unknown'
  postsDir: string        // 文章所在目录，如 '_posts'
  i18nStrategy: {
    type: 'directory'     // /{lang}/_posts/xxx.md
    pattern: string       // '/{lang}/_posts/{file}'
  }
}
```

**Jekyll i18n 文件策略**：

- 翻译文件放在 `/{lang}/_posts/xxx.md`
- 例：`_posts/2025-01-01-hello.md` → `en/_posts/2025-01-01-hello.md`

### 7.2 AI 翻译引擎 (translator.ts)

使用 `openai` npm 包，切换 baseURL 即可对接不同 provider。

```typescript
import OpenAI from 'openai'

const client = new OpenAI({
  apiKey: config.aiApiKey,
  baseURL: getBaseURL(config.aiProvider) // 根据 provider 返回对应 URL
})
```

**翻译 Prompt 设计**：

```
System:
你是一个专业的博客文章翻译器。将以下 Markdown 文章从 {source_lang} 翻译成 {target_lang}。

规则：
1. 保留所有 Markdown 格式（标题、列表、粗体、斜体、链接、图片等）
2. 保留所有代码块（```...```）内容不翻译
3. 保留所有 HTML 标签
4. 保留图片路径不变
5. Front matter 中仅翻译 title、description、summary、tags 字段，其余保持原样
6. 保留原文中的专有名词、品牌名、技术术语的英文形式
7. 直接输出翻译结果，不要添加任何解释

User:
{article_content}
```

**并发控制**：

- 使用 `Promise.allSettled` 并发翻译多篇文章
- 默认并发数 3（可配置），避免 API 速率限制
- 单篇失败不影响其他文章，失败记录到日志

**错误重试**：

- API 调用失败重试 3 次，指数退避（1s, 2s, 4s）
- 429 (Rate Limit) 自动等待 `Retry-After` 头指定的时间

### 7.3 翻译文件生成 (generator.ts)

**输入**：翻译后的内容 + 框架检测结果
**输出**：写入对应路径的文件

Jekyll 规则：


| 原文路径                     | 翻译文件路径                    |
| ------------------------------ | --------------------------------- |
| `_posts/2025-01-01-hello.md` | `en/_posts/2025-01-01-hello.md` |
| `_posts/2025-01-01-hello.md` | `ja/_posts/2025-01-01-hello.md` |

生成时在 front matter 中追加 `lang` 字段：

```yaml
---
title: "Hello World"
lang: en
permalink: /en/posts/hello/
---
```

### 7.4 语言切换链接注入 (lang-link.ts)

在 front matter 之后、正文之前插入：

```markdown
> 🌐 [English](/en/posts/hello/) | [中文](/posts/hello/) | [日本語](/ja/posts/hello/)
```

**逻辑**：

1. 用正则 `^>\s*🌐\s*\[.+\]\(.+\)` 检测是否已有语言链接
2. 已有 → 替换整行（可能新增了语言）
3. 没有 → 在 front matter `---` 结束后的第一行插入
4. 原文和所有翻译文件都需要注入/更新

**链接生成**：根据框架 i18n 策略和文章 permalink 生成各语言的 URL。

### 7.5 增量检测 (diff.ts + state.ts)

**state.json 结构**（存放于 `.blog-global/state.json`）：

```json
{
  "version": 1,
  "files": {
    "_posts/2025-01-01-hello.md": {
      "hash": "sha256:abc123...",
      "translated": {
        "en": "sha256:def456...",
        "ja": "sha256:ghi789..."
      }
    }
  }
}
```

**增量检测流程**：

1. 读取 `state.json`（不存在则视为全量翻译）
2. 遍历 `_posts/` 下所有 `.md` 文件
3. 计算每个文件的 SHA-256 hash
4. 对比 state：
   - hash 不存在 → 新文件，全文翻译
   - hash 变化 → 修改文件，全文重新翻译
   - hash 未变 → 跳过
5. state 中有但文件已删除 → 删除对应翻译文件
6. 检查 `.blog-global-ignore` 过滤

**注意**：不依赖 `git diff`（CLI 可能不在 git 环境），改用 hash 对比，更通用。

### 7.6 Git/PR 操作 (git-pr.ts)

**Action 环境下**（使用 `@actions/github` 提供的 Octokit 实例）：

```typescript
import * as github from '@actions/github'

const octokit = github.getOctokit(githubToken)
```

流程：

1. 创建分支 `blog-global/translate-{timestamp}`
2. 将翻译文件 + 更新后的原文（含语言链接）提交到新分支
3. 使用 Octokit `pulls.create` 创建 PR

```typescript
await octokit.rest.pulls.create({
  owner, repo,
  title: '[blog-global] Auto translation update',
  head: branchName,
  base: 'main',
  body: generatePRBody(translatedFiles)
})
```

**CLI 环境下**：

- 使用 `child_process.execSync` 调用原生 git 命令
- `git checkout -b`、`git add`、`git commit`、`git push`
- 然后通过 Octokit REST API 创建 PR（需要用户提供 GitHub Token）

**权限要求**：

- Action + GITHUB_TOKEN：workflow 需声明 `permissions: { contents: write, pull-requests: write }`
- Action + GitHub App：App 需授予 `Contents: Read & Write` + `Pull requests: Read & Write`
- CLI：需要 GitHub Personal Access Token（scope: repo）或 GitHub App private key

### 7.7 Ignore 解析 (ignore.ts)

使用 `ignore` npm 包解析 `.blog-global-ignore` 文件。

```typescript
import ignore from 'ignore'

const ig = ignore().add(fs.readFileSync('.blog-global-ignore', 'utf8'))
const shouldTranslate = !ig.ignores(filePath)
```

## 8. 数据流

```
[触发] push 事件 / CLI 执行
    │
    ▼
[1] 检测框架 → jekyll
    │
    ▼
[2] 扫描 _posts/*.md → 文件列表
    │
    ▼
[3] 过滤 .blog-global-ignore
    │
    ▼
[4] 读取 state.json → 增量对比 → 需翻译文件列表
    │
    ▼
[5] 并发调用 AI 翻译（每文件 × 每目标语言）
    │
    ▼
[6] 生成翻译文件到 /{lang}/_posts/
    │
    ▼
[7] 注入语言切换链接（原文 + 翻译文件）
    │
    ▼
[8] 更新 state.json
    │
    ▼
[9] 创建分支 → 提交 → 创建 PR
```

## 10. 构建与发布

### Action 构建

```bash
npm run build    # tsc 编译
npm run package  # ncc 打包到 dist/index.js
```

`dist/index.js` 需提交到仓库（GitHub Action 直接运行此文件）。

### CLI 发布

`package.json` 中配置 `bin` 字段：

```json
{
  "name": "blog-global",
  "bin": {
    "blog-global": "./dist/cli.js"
  }
}
```

发布到 npm：`npm publish`

**注意**：Action 入口和 CLI 入口是两个独立的编译产物：

- `dist/index.js` → Action（由 ncc 打包，包含所有依赖）
- `dist/cli.js` → CLI（npm 安装时依赖随 node_modules 安装）

### package.json scripts

```json
{
  "scripts": {
    "build": "tsc",
    "package": "ncc build src/action.ts -o dist",
    "package:cli": "tsc && chmod +x dist/cli.js",
    "prepare": "npm run package"
  }
}
```
