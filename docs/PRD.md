# blog-global 需求规格文档 (PRD)

## 1. 项目概述

**项目名称：** blog-global

**一句话描述：** 一个开源的 GitHub Pages 静态博客自动翻译工具，基于 AI 将博客文章翻译成多种语言，并自动插入语言切换链接。

**目标用户：** 使用 GitHub Pages 托管静态博客的开发者

**核心价值：** 零成本让个人博客支持多语言，增加国际流量

## 2. MVP 功能范围

### 2.1 核心功能

| #  | 功能         | 描述                                                                             |
| -- | ------------ | -------------------------------------------------------------------------------- |
| F1 | 仓库分析     | 输入`xxx.github.io` 仓库地址，自动识别博客框架（Jekyll/Hugo/Hexo等）和文章位置 |
| F2 | 文章翻译     | 通过 AI 网关（OpenRouter/硅基流动）将文章翻译成目标语言，保留 Markdown/HTML 格式 |
| F3 | 文件生成     | 根据博客框架的 i18n 规范生成翻译后的文件，放置在正确位置                         |
| F4 | 语言切换链接 | 在原文和翻译页面中插入语言切换链接（文字链接：EN\| 中文 \| 日本語）              |
| F5 | 增量翻译     | 基于 Git diff 检测变更，只翻译新增/修改的文章                                    |
| F6 | 自动 PR      | 翻译完成后自动创建 Pull Request，用户审核后合并                                  |

### 2.2 不在 MVP 范围内

- Web 管理界面
- 轮询 URL 检测变更
- 翻译非 GitHub Pages 的网站
- 翻译非文章内容（导航、footer 等）
- AI 风格匹配的语言切换组件
- 翻译记忆/术语表

## 3. 翻译文件存放规范

根据框架不同，遵循各自的 i18n 惯例：

| 框架   | 结构                                                                 |
| ------ | -------------------------------------------------------------------- |
| Jekyll | `/{lang}/_posts/xxx.md` 或使用 front matter 标记语言               |
| Hugo   | `/content/{lang}/posts/xxx.md` 或 `/content/posts/xxx.{lang}.md` |
| Hexo   | `/source/_posts/{lang}/xxx.md`                                     |

## 4. 工作流程图

```
用户 push 文章 → GitHub Action 触发
                       ↓
              检测变更的文章文件
                       ↓
              识别框架和 i18n 规范
                       ↓
              AI 翻译变更内容
                       ↓
              生成翻译文件 + 插入语言链接
                       ↓
              创建 PR → 用户审核合并
```

## 6. 语言切换链接

MVP 采用简单文字链接，插入位置在文章顶部（front matter 下方）：

```markdown
> 🌐 [English](/en/posts/my-post/) | [中文](/posts/my-post/) | [日本語](/ja/posts/my-post/)
```

插入逻辑：

1. 检测文章是否已有语言切换链接（通过正则匹配）
2. 没有则在文章开头插入
3. 已有则更新链接列表（如新增了语言）

## 7. 忽略配置

支持 `.blog-global-ignore` 文件（类似 `.gitignore` 语法），列在其中的文章文件不会被翻译：

```
# 不翻译的文章
_posts/2024-01-01-test.md
_posts/2025-03-15-private-note.md
```

匹配到的文章即使内容发生变更也不会触发翻译。

## 8. 增量翻译策略

1. 通过 `git diff` 获取自上次翻译以来变更的文件列表
2. 对于新增文件：全文翻译
3. 对于修改文件：全文重新翻译（MVP阶段，段落级增量翻译复杂度过高）
4. 对于删除文件：删除对应的翻译文件
5. 使用 `.blog-global/state.json` 记录翻译状态（文件hash → 已翻译语言）

## 9. 非功能性需求

| 项目     | 要求                                           |
| -------- | ---------------------------------------------- |
| 翻译质量 | 保留所有 Markdown 格式、代码块、链接、图片路径 |
| 性能     | 支持并发翻译多篇文章                           |
| 错误处理 | 单篇翻译失败不影响其他文章，失败可重试         |
| 幂等性   | 重复运行不会产生重复内容                       |
| 安全性   | API Key 通过环境变量/secrets 注入，不存入代码  |

## 10. MVP 里程碑

| 阶段 | 内容                                                         |
| ---- | ------------------------------------------------------------ |
| M1   | 项目架构搭建                                                 |
| M2   | 仓库分析器（克隆仓库、识别博客框架、定位文章文件）           |
| M3   | 翻译引擎（OpenRouter/硅基流动接入、Markdown格式保持）        |
| M4   | 文件生成器（按框架i18n规范生成翻译文件）                     |
| M5   | 语言切换链接注入（检测、插入、更新链接）                     |
| M6   | 增量翻译（Git diff检测变更、state.json状态管理、ignore过滤） |
| M7   | Git操作 + 自动创建PR                                         |
| M8   | GitHub Action封装 + CLI打包发布                              |
