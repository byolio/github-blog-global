# 快速启动说明文档

本文档已按实际跑通的流程验证（本地 mock 测试 + 硅基流动真实 API + 真实 GitHub 仓库创建 PR），照着做即可。

## 1. 本地 CLI 模式

### 1.1 构建

```bash
cd blog-global
npm install
npm run build          # tsc 编译到 dist/
npm run package:cli    # ncc 打包出可执行的 dist/cli.js
```

### 1.2 交互式运行

在**你的博客仓库根目录**下执行（不是在 blog-global 项目目录里）：

```bash
node /path/to/blog-global/dist/cli.js
```

依次交互式输入：AI Provider → API Key → 翻译模型 → 源语言 → 目标语言（多选，会校验是否在支持列表里）→ PR 目标分支 → 是否自动创建 PR → （创建 PR 时）GitHub Token。

### 1.3 非交互模式（脚本/自动化用）

```bash
node /path/to/blog-global/dist/cli.js \
  --provider siliconflow \
  --api-key $BLOG_GLOBAL_API_KEY \
  --model deepseek-ai/DeepSeek-V3 \
  --base-lang zh-CN \
  --target-langs en,ja \
  --create-pr \
  --token $GITHUB_TOKEN \
  --base-branch main \
  --site-url https://www.byolio.blog
```

`--site-url` 可选。不填时语言切换链接是站内相对路径（如 `/en/posts/xxx/`）；填了会生成完整绝对地址（如 `https://www.byolio.blog/en/posts/xxx/`），中文页固定是 `{site-url}/posts/xxx/`，其他语言固定是 `{site-url}/{lang}/posts/xxx/`，不用带协议头也行（会自动补 `https://`）。Action 里对应的输入是 `site_url`，配置文件里对应 `site_url` 字段。

也可以把 `--api-key` 换成环境变量 `BLOG_GLOBAL_API_KEY`，`--token` 换成 `GITHUB_TOKEN`。

注意（已修复过的关键行为）：

- **语言代码非法会直接报错退出**（exit 1），不会静默跳过或崩溃，错误信息会列出所有支持的语言代码。
- **非交互模式下，如果创建 PR 失败（token 无效、分支不存在等），会直接打印错误并退出（exit 1），不会卡在交互菜单等你输入**——这是专门为脚本/CI 场景改的。已经翻译好的文件和 `.blog-global/state.json` 会保留在本地磁盘上，不会丢失，下次重新跑 `--create-pr` 即可续上（失败的语言会自动重新翻译，不会因为文章内容没变就被跳过）。
- 只有**交互模式**（不带参数直接跑 `node dist/cli.js`）在 PR 创建失败时才会弹出"重试/换 Token/跳过"的菜单。

### 1.4 进度输出

```text
[Progress] 1/5 files translated (AI tasks: 2/10 using deepseek-ai/DeepSeek-V3)
```

## 2. GitHub Action 模式

### 2.1 打包并提交

```bash
npm run build
npm run package      # 生成 dist/index.js（Action 用）
npm run package:cli  # 生成 dist/cli.js（可选，CLI 用）
git add dist action.yml
git commit -m "build: update dist"
git push
```

`dist/` 已经不在 `.gitignore` 里了，正常 `git add` 就会带上，**这一步是 Action 能被别人 `uses:` 引用的前提，漏了这步 Action 会直接报找不到 `dist/index.js`**。

### 2.2 博客仓库里配置 workflow

在你的博客仓库（不是 blog-global 仓库本身）创建 `.github/workflows/translate.yml`：

#### 最简配置（默认 GITHUB_TOKEN，够用，缺点是不会触发下游 workflow）

```yaml
name: Blog Global Translate
on:
  push:
    branches: [main]
    paths: ['_posts/**']

permissions:
  contents: write
  pull-requests: write

jobs:
  translate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0   # 增量检测需要完整历史

      - uses: 你的用户名/blog-global@main
        with:
          ai_provider: siliconflow
          ai_api_key: ${{ secrets.AI_API_KEY }}
          ai_model: deepseek-ai/DeepSeek-V3
          base_lang: zh-CN
          target_langs: en,ja
```

需要在博客仓库 Settings → Secrets and variables → Actions 里加一个 `AI_API_KEY`。

#### 用自己的 PAT（想让创建的 PR 能触发别的 workflow）

```yaml
      - uses: 你的用户名/blog-global@main
        with:
          ai_provider: siliconflow
          ai_api_key: ${{ secrets.AI_API_KEY }}
          ai_model: deepseek-ai/DeepSeek-V3
          base_lang: zh-CN
          target_langs: en,ja
          github_token: ${{ secrets.PAT_TOKEN }}
```

`PAT_TOKEN` 对应《人工进行的配置文档》里方案 B 生成的 Token。

## 3. 验证清单

- [ ]  博客仓库有 `_config.yml` + `_posts/`，且已推送到远端
- [ ]  AI Provider / API Key / 模型名 三者都对得上（模型名要带完整组织路径）
- [ ]  `--create-pr` / Action 触发后，能在仓库看到新分支 `blog-global/translate-<时间戳>` 和对应的 PR
- [ ]  PR 的 "Files changed" 里能看到 `_posts/xxx.md`（modified，插入了语言链接）、`{lang}/_posts/xxx.md`（added，翻译文件）、`.blog-global/state.json`（added/modified）
- [ ]  语言切换链接（`> 🌐 [中文](...) | [English](...)`）里的路径是相对路径，直接在 GitHub 网页点击会报错是正常的——要等 GitHub Pages 部署后才能真正跳转
